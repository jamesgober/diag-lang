# diag-lang v0.2.0 — Diagnostic model & the caret renderer

**The hard part, front-loaded.** v0.2.0 turns the scaffold into a working
diagnostics layer. It introduces the `Severity`, `Label`, and `Diagnostic` model
and — the genuinely hard part the roadmap put first on purpose — the renderer that
takes a labelled span and its source and prints it with a caret underline aligned
exactly under the offending characters. Alignment over multi-byte UTF-8 and tab
expansion is where naive renderers break, so it is proven here, byte for byte,
before any of the easier multi-label work is built on top of it.

## What is diag-lang?

The diagnostics layer of a compiler front-end. It collects diagnostics — a
severity, a message, and a labelled span — and renders them with source context:
the offending line with a caret underlining exactly where the problem is, in the
style developers expect from a modern toolchain. It consumes positions from
`span-lang` and source text from `source-lang`, and it owns presentation only: a
lexer, parser, or type checker decides *what* is wrong, diag-lang decides how it
looks.

## What's new in 0.2.0

### The diagnostic model — `Severity`, `Label`, `Diagnostic`

A `Diagnostic` is the unit a compiler stage emits: a `Severity` (`Error`,
`Warning`, `Note`, `Help`), a headline message, and a primary `Label`. The primary
label is supplied at construction, so a diagnostic always knows where it points —
there is no half-built state and the renderer has no "no location" branch. A
`Label` is a `Span` paired with a message; an `unlabelled` label points without
annotating, drawn as a bare caret.

```rust
use diag_lang::{Diagnostic, Label, Severity, Span};

let diag = Diagnostic::new(
    Severity::Error,
    "type mismatch",
    Label::new(Span::new(8, 11), "expected `Int`, found `Str`"),
);
```

Construction is on the front-end's hot path, so it allocates only what the message
and label text require and holds a single `Copy` span rather than anything heavier.

### The renderer — caret alignment, proven

`Renderer::render` resolves the span against a `SourceMap`, finds the file and
line, and prints the line with a caret run underneath:

```text
error: cannot find value `foo` in this scope
 --> main.rs:2:13
  |
2 |     let x = foo();
  |             ^^^ not found in this scope
  |
```

The alignment is computed in **display columns**, not bytes. A multi-byte UTF-8
character counts as one column, so a five-character name like `héllo` (six bytes)
draws five carets, not six. A tab expands to the next four-column stop, and the
displayed line shows the expanded spaces, so the carets line up under the source
whatever the reader's terminal tab width. A zero-width span still draws a single
caret as a pointer. `render_to` writes into any `fmt::Write` sink, so a caller can
render into a buffer it already owns instead of allocating per diagnostic.

### A defined fallback, never a panic

Rendering hostile or stale input is a defined outcome. A span past the end of the
sources, or one referencing an offset no source covers, renders the header and a
note rather than panicking:

```text
error: unexpected end of input
 = note: span 10..12 is outside the loaded sources
```

### `span-lang` and `source-lang`, wired and used

The two dependencies are now real, not reserved. A `Label`'s span is a
`span-lang` `Span`, and the renderer resolves it through a `source-lang`
`SourceMap` — `locate` to find the file, the per-source `LineIndex` to find the
line and column. Both `Span` and `SourceMap` are re-exported from the crate root,
so a downstream building and rendering diagnostics need not name either dependency
directly.

### Snapshot and property tests

`tests/rendering.rs` holds the section-4 invariants to byte-exact expected output:
ASCII, multi-byte prefixes and spans, tab expansion, zero-width and unlabelled
spans, multi-digit line gutters, multi-file maps, and the out-of-range fallback.
`tests/properties.rs` cross-checks the caret width against a deliberately naive
column walk over a wide `proptest` input space, and holds that rendering is total
(never panics, always newline-terminated) and deterministic.

## Breaking changes

**None** to released API: v0.1.0 had no public surface. The reserved, no-op
`serde` feature is removed rather than carried unused toward the frozen 1.0
surface; it can return post-1.0 as a non-breaking minor if a need appears.
